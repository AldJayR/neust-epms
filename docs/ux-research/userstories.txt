Now we can create user stories properly because we have already gone through:

**Use Cases → Empathy Maps → Personas → Journey Maps → Pain Points → User Stories**

The mistake would be to mechanically convert every use-case step into:

> As a user, I want to upload a file so that the file is uploaded.

That produces feature-centered stories.

Instead, the stories should come from the **goals and pain points we identified**, while the existing use cases act as constraints and evidence about the intended workflow. 

# User Stories: Faculty / Project Leader

## Epic: Understand and Navigate My Project Journey

**US-F01 — View the project lifecycle**

> As a Faculty Project Leader, I want to see the complete lifecycle of my project so that I can understand where it currently is and what will happen next.

**US-F02 — Understand the current status**

> As a Faculty Project Leader, I want to see what my project's current status means so that I do not have to interpret administrative terminology.

**US-F03 — Know who currently owns the action**

> As a Faculty Project Leader, I want to know who is currently responsible for the next action so that I can understand whether I need to act or wait.

**US-F04 — Know my next action**

> As a Faculty Project Leader, I want the system to clearly show what I need to do next so that I can move my project forward without asking someone for instructions.

**US-F05 — View project history**

> As a Faculty Project Leader, I want to see the history of submissions, reviews, decisions, and status changes so that I can understand how my project reached its current state.

These stories directly address the largest journey pain point: **the system understands the workflow, but the user may not**.

---

## Epic: Prepare a Complete Proposal

**US-F06 — View proposal requirements**

> As a Faculty Project Leader, I want to see all proposal requirements before I begin submitting so that I can prepare the correct documents.

**US-F07 — Access official templates**

> As a Faculty Project Leader, I want to access the current official templates so that I do not accidentally use outdated documents.

**US-F08 — Save proposal progress**

> As a Faculty Project Leader, I want to save an incomplete proposal as a draft so that I can prepare it across multiple work sessions.

**US-F09 — See submission completeness**

> As a Faculty Project Leader, I want to see which requirements are complete or missing so that I can correct problems before submission.

**US-F10 — Review before submitting**

> As a Faculty Project Leader, I want to review my proposal package before final submission so that I can verify that I am sending the correct information and files.

Notice the shift from UC-4. The use case begins at **submission**. 

The user stories reveal that the actual user experience begins at **preparation**.

---

## Epic: Submit and Track My Proposal

**US-F11 — Receive submission confirmation**

> As a Faculty Project Leader, I want immediate confirmation that my proposal was successfully submitted so that I know it entered the review process.

**US-F12 — See where my proposal is**

> As a Faculty Project Leader, I want to see the current review stage of my proposal so that I do not need to manually follow up for routine status updates.

**US-F13 — Know how long my proposal has been waiting**

> As a Faculty Project Leader, I want to see when my proposal entered its current stage so that I can understand how long it has been waiting.

**US-F14 — Be notified of meaningful changes**

> As a Faculty Project Leader, I want to be notified when my proposal requires my action or moves to a significant new stage so that I do not need to constantly check the system.

This addresses the **invisible waiting** pain point.

---

## Epic: Respond to Proposal Feedback

**US-F15 — Understand why my proposal was returned**

> As a Faculty Project Leader, I want to see clear feedback when my proposal is returned so that I know why revision is required.

**US-F16 — Know what must be revised**

> As a Faculty Project Leader, I want feedback to identify what needs to be changed so that I can revise the correct parts of my submission.

**US-F17 — Access previous submissions**

> As a Faculty Project Leader, I want to access previous versions of my proposal so that I can understand the history of my revisions.

**US-F18 — Submit a revised proposal**

> As a Faculty Project Leader, I want to submit revised documents against the returned proposal so that my project can continue through the existing review process.

**US-F19 — Know where my revision goes next**

> As a Faculty Project Leader, I want to know who will review my revised proposal after resubmission so that I understand the next stage of the process.

That final story is directly supported by the bypass logic in UC-4 and UC-6.

## Epic: Prepare My Approved Project for Implementation

**US-F20 — Distinguish approval from activation**

> As a Faculty Project Leader, I want to know whether my project is merely approved or officially authorized for implementation so that I do not begin prematurely.

**US-F21 — See activation requirements**

> As a Faculty Project Leader, I want to see all requirements that must be completed before activation so that I understand what is preventing implementation.

**US-F22 — See the current blocker**

> As a Faculty Project Leader, I want the system to identify the exact requirement blocking activation so that I know why my approved project cannot proceed.

**US-F23 — Know who is responsible for each requirement**

> As a Faculty Project Leader, I want to see who is responsible for each incomplete activation requirement so that I know whether I need to act or wait.

This is one of the strongest story groups because it directly addresses the dangerous confusion between **approved** and **ongoing**.

# Cross-role stories

Some of the most important stories should not belong to only one persona.

**US-X01 — Meaningful notifications**

> As a system user, I want notifications to tell me what happened and whether I need to act so that I can respond appropriately.

**US-X02 — Clear next actions**

> As a system user, I want each workflow item to clearly identify the next required action so that I do not have to infer what happens next.

**US-X03 — Role-appropriate visibility**

> As a system user, I want to see only the projects and actions relevant to my responsibilities so that I can focus on my work.

**US-X04 — Preserve history**

> As a system user, I want important submissions, decisions, and status changes to remain traceable so that the project history can be understood later.

---

The important result is that these user stories do **not** map one-to-one with the nine use cases.

That is a good sign.

The use cases describe **system behavior**. The user stories reveal **human needs**. For example, UC-7 says the system detects an overdue report; the user stories reveal that faculty need warnings *before* it becomes overdue. UC-4 says the user submits documents; the stories reveal that the user needs guidance *before* submission. UC-3 says the system displays metrics; the stories reveal that RET Chairs and Director/Admin users first need to know *what requires action*. 

The next useful step is to **prioritize these stories into MVP, Should Have, Could Have, and Later**. We currently have more than enough stories to overbuild the system, so prioritization is where the Design Thinking work starts protecting your scope.
